# 用户信息显示修复测试记录

## 问题描述

**问题现象：**
- 新注册的李四账号登录成功后，DID信息仍然显示张三的个人信息
- 包括姓名、手机号码、邮箱地址、角色、年龄等所有字段都显示张三的硬编码信息
- 页面顶部的用户信息也显示硬编码的"Ericson Lau"和"kiringamcongmall.com"

**问题分析：**
通过代码审查发现，在`showUserDetail()`函数和`displayUserInfo()`函数中使用了硬编码的默认值，而不是从localStorage中获取真实的用户信息。

## 修复过程

### 1. 问题定位
- 检查`file-management.js`中的`showUserDetail()`函数
- 发现用户详情显示使用了硬编码的默认值：
  ```javascript
  // 修复前（错误）
  <span class="value">${userInfo.name || '张三'}</span>
  <span class="value">${userInfo.phone || '13748677343'}</span>
  <span class="value">${userInfo.email || '1393566147@qq.com'}</span>
  <span class="value">${userInfo.role || '业主'}</span>
  <span class="value">${userInfo.age || '18'}</span>
  ```

### 2. 修复方案
将硬编码的默认值改为动态获取用户信息，支持多种字段名格式：

```javascript
// 修复后（正确）
<span class="value">${userInfo.Name || userInfo.name || '未知'}</span>
<span class="value">${userInfo.Phone || userInfo.phone || '未知'}</span>
<span class="value">${userInfo.Email || userInfo.email || '未知'}</span>
<span class="value">${userInfo.Role || userInfo.role || '未知'}</span>
<span class="value">${userInfo.Age || userInfo.age || '未知'}</span>
```

### 3. 修复内容

#### 3.1 用户详情显示修复
- **基本信息部分**：姓名、手机号码、邮箱地址、角色、年龄
- **DID身份信息**：DID标识符、注册时间
- **区块链信息**：交易哈希

#### 3.2 页面顶部用户信息修复
- 用户名显示：从`userData.Name`或`userData.name`获取
- 邮箱显示：从`userData.Email`或`userData.email`获取

#### 3.3 字段名兼容性
支持多种字段名格式，确保与后端API返回的数据结构兼容：
- `Name/name`：用户姓名
- `Phone/phone`：手机号码
- `Email/email`：邮箱地址
- `Role/role`：用户角色
- `Age/age`：用户年龄
- `DID/did`：DID标识符
- `RegisterTime/registerTime`：注册时间
- `Txid/txid`：交易哈希

## DID 上链存证调试与修复记录（Fabric 2.5.5 + GoFrame）

### 问题现象
- SDK 侧 DID 登记接口（/api/did/register）返回 500：failed to collect enough transaction endorsements / failed to endorse transaction。
- 同期 peer CLI 通过显式指定双组织背书（--peerAddresses 7051/9051）可以成功登记与查询。

### 根因分析
- Fabric 通道上 `basic` 链码的背书策略为 Org1+Org2，SDK 需收集双组织背书。
- 期间多次出现网络与节点状态问题：
  - `peer0.org2.example.com` 容器退出（Exited 255），宿主机 9051 未可用。
  - docker compose 未设置 `DOCKER_SOCK` 或网络标签不匹配，导致容器无法正常启动/加入网络。
  - /etc/hosts 缺少 `peer0.org1.example.com / peer0.org2.example.com / orderer.example.com` 映射，服务发现返回的主机名在宿主机不可解析。
- SDK 网关初始化与调用路径未严格对齐官方推荐，导致在服务发现返回不可达端点时收集背书失败。

### 修复步骤（按时间线归纳）
1) 恢复网络与容器
   - 清理与重建 docker compose 网络（修复标签不匹配）：
     ```bash
     docker network rm fabric_test || true
     export DOCKER_SOCK=/var/run/docker.sock
     docker compose -p test \
       -f compose/compose-test-net.yaml \
       -f compose/docker/docker-compose-test-net.yaml \
       up -d orderer.example.com peer0.org1.example.com peer0.org2.example.com
     ```
   - 确认端口联通：7050/7051/9051 可达。
   - /etc/hosts 增加映射：`127.0.0.1 peer0.org1.example.com peer0.org2.example.com orderer.example.com`。

2) 重新部署链码（必要时）
   - 使用测试网络脚本部署 `basic` 链码（包含 DID 功能）：
     ```bash
     ./network.sh deployCC -ccn basic -ccp /root/home/tmp/basic/src/src -ccl go -ccv 1.0 -ccs 1
     ```
   - 若已有定义，升级遵循“当前序列 + 1”。
   - 用 CLI 验证双背书登记/查询成功（确保链路健康）。

3) SDK 网关与调用路径修正（严格按官方 Gateway 流程）
   - 文件：`internal/service/chaincode.go`
   - 网关初始化（InitChaincodeService）
     - 采用 TLS 直连 Org1 Peer `localhost:7051`（与 CLI 一致，连接稳定）。
     - 使用 Org1 Admin 证书与私钥，创建 Gateway → Network("mychannel") → Contract("basic")。
   - DID 登记调用（RegisterDIDOnChain）
     - 改为与其他写入一致的 `contract.SubmitTransaction("RegisterDID", ...)`，统一调用路径，避免服务发现差异带来的不确定性。
     - 失败场景定位通过提升日志级别（GF_LOG_LEVEL/FABRIC_LOGGING_SPEC/GRPC_GO_*）。

4) 接口验证
   - 登记：`/api/did/register`（body: email, addresses, phone, password, info）
   - 验证：`/api/did/verify`（body: email, password）
   - 返回包含 `did/txID/timestamp`（登记）与 `valid/message`（验证）。

### 关键命令备忘
```bash
# 网络：启动基础节点
export DOCKER_SOCK=/var/run/docker.sock
docker compose -p test -f compose/compose-test-net.yaml -f compose/docker/docker-compose-test-net.yaml \
  up -d orderer.example.com peer0.org1.example.com peer0.org2.example.com

# 链码：首次部署
./network.sh deployCC -ccn basic -ccp /root/home/tmp/basic/src/src -ccl go -ccv 1.0 -ccs 1

# CLI 验证登记（双组织背书）
peer chaincode invoke -o localhost:7050 --ordererTLSHostnameOverride orderer.example.com --tls --cafile $ORDERER_CA \
  --channelID mychannel --name basic \
  --peerAddresses localhost:7051 --tlsRootCertFiles $PEER0_ORG1_CA \
  --peerAddresses localhost:9051 --tlsRootCertFiles $PEER0_ORG2_CA \
  -c '{"Args":["RegisterDID","user@example.com","A,B","13800138000","Passw0rd!","info"]}'
```

### 结果与结论
- 最终 CLI 与 SDK 均可稳定完成 DID 上链登记与验证；SDK 登记返回 `did/txID/timestamp`，验证返回 `valid:true`。
- 读操作（EvaluateTransaction）不需双背书，天然更稳定；写操作统一走 SubmitTransaction 后亦稳定，前提是双组织 peer 可达。

### 影响与建议
- 背书策略仍为 Org1+Org2，未降低安全性；所有写交易（包含分片存储写入）路径保持一致，更易维护。
- 若后续仍遇“背书不足”，优先排查：
  - `peer0.org2.example.com` 容器状态与 9051 端口；
  - /etc/hosts 主机名解析；
  - 通道上的链码定义版本/序列与两组织批准状态；
  - SDK 服务日志（建议在生产保留 INFO 级别）。
